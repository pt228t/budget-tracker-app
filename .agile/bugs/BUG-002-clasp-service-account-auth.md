# Bug Ticket: BUG-002 - Clasp Push Fails with Service Account Auth

## Issue Description
Automated deployments to Google Apps Script failed in the GitHub Actions pipeline (`clasp push`) when using a GCP Service Account. 

**Error messages observed:**
1. `Permission 'iam.serviceAccounts.getAccessToken' denied on resource`
2. `The incoming JSON object does not contain a client_id field` (when using JWT directly)
3. `User has not enabled the Apps Script API. Enable it by visiting https://script.google.com/home/usersettings then retry.`

## Root Cause
While Google APIs generally support Service Accounts via JWT exchange, the **Apps Script API** requires an explicit, user-level API enablement via the Google Apps Script dashboard UI (`https://script.google.com/home/usersettings`). 

Because a Service Account is a headless robot account, it has no UI access and cannot toggle this setting. Therefore, a standard GCP Service Account *cannot* be used with `@google/clasp` for personal Apps Script projects without paid Workspace features (like Domain-Wide Delegation).

## Resolution
1. **Pivoted Authentication Strategy:** Switched the CI/CD pipeline from using a Service Account JSON Key to using a **Refresh Token** generated by the project owner.
2. **Environment Variables:** Replaced `GOOGLE_SERVICE_ACCOUNT_JSON` with three new secrets sourced from the developer's local `~/.clasprc.json`:
   - `CLASP_REFRESH_TOKEN`
   - `CLASP_CLIENT_ID`
   - `CLASP_CLIENT_SECRET`
3. **Deploy Script Update:** Updated `scripts/deploy-apps-script.mjs` to dynamically write a complete `~/.clasprc.json` on the GitHub Runner. 
4. **Auto-refresh Behavior:** `clasp` automatically uses the injected refresh token to request a short-lived access token directly from Google OAuth2, allowing successful deployments.
5. **Documentation:** Updated `APPS_SCRIPT_SETUP.md` to explain the new Refresh Token workflow and removed all references to the Service Account method.

## Status
✅ Resolved (Pipeline successfully deploying via refresh token)
